# M001/S02: Full review lifecycle parity — Research

**Date:** 2026-03-21

## Summary

S02 primarily owns **R004** (preserve the review lifecycle contract) and also advances **R001** and **R010** by expanding the standalone runtime from create/claim/proposal basics into verdict, discussion, activity, close/requeue, and counter-patch behavior. The main planning risk is that the current TypeScript contract frozen in S01 is **not yet semantically aligned** with the Python broker: S01 uses `submitted` instead of Python `in_review`, allows `closed` from most states, and has no shared payloads yet for verdicts, messages, activity, or counter-patch decisions.

The repo already has the right seams for this slice. `review-broker-core` is the canonical contract layer, `review-broker-server` already owns durable state + audit + notifications, and the legacy Python broker plus its tests provide concrete behavior to port. The highest-value approach is to treat the Python tests as behavior specs, but implement them in the project’s existing Vitest style, per the loaded `test` skill’s rule to match existing test conventions rather than invent a new testing style.

## Recommendation

Start S02 by reconciling the **shared TS contract and state machine** before adding server logic. Following the loaded `debug-like-expert` skill’s “verify, don’t assume” rule, do not implement verdict/discussion code on top of the current S01 transition table until the planner explicitly decides how to map legacy Python semantics onto the TS vocabulary.

Recommended mapping: **keep the existing TS `submitted` status as the S02 equivalent of Python `in_review`**, instead of reintroducing a brand-new status string. That preserves S01’s typed surface while still allowing parity behavior: reviewer comments can move a claimed review into `submitted`, final verdicts can move `submitted -> approved|changes_requested`, proposer follow-up after `changes_requested` can requeue to `pending`, and close can be tightened to the legacy-approved path. This is less disruptive than re-opening every S01 contract consumer to rename statuses, but still requires updating `packages/review-broker-core/src/state-machine.ts` and its tests so the transition table matches actual lifecycle intent.

Following the loaded `test` and `review` skills, implementation should stay contract-first and test-heavy:
- extend shared schemas in `review-broker-core` first
- add a **new migration file** (never edit `001_init.sql`, because `open-database.ts` checksum-tracks applied migrations)
- add repository support for message/discussion/counter-patch fields
- then implement broker-service methods and parity tests modeled on the Python broker tests

## Constraints

- `packages/review-broker-server/src/db/open-database.ts` checksum-tracks each `.sql` migration file. **Editing `001_init.sql` will break reopen behavior**; S02 must add a new migration file instead.
- `packages/review-broker-server/test/sqlite-bootstrap.test.ts` and `packages/review-broker-server/test/start-broker.smoke.test.ts` currently hard-code a single migration. They will fail until updated alongside the new migration.
- `packages/review-broker-server/src/db/migrations/001_init.sql` created a minimal `messages` table (`author_id`, `body`, `created_at`) that is not rich enough for legacy discussion semantics. The planner should assume a schema extension is required, not just service wiring.
- Current S01 core tests already freeze the TS vocabulary around `submitted`. If the planner wants literal Python `in_review` naming, it must intentionally reopen and update the S01 contract/tests rather than accidentally drifting into mixed terminology.
- The project has **no active LSP server** in this harness, so file-level planning should assume direct file reads/targeted grep rather than semantic rename support during execution.

## Common Pitfalls

- **Editing `001_init.sql` instead of adding `002_*.sql`** — this will trigger migration checksum mismatch in `open-database.ts` and break restart-safe proof.
- **Implementing lifecycle shapes only in `broker-service.ts`** — S04 depends on shared contracts; if verdict/discussion payloads are not added to `review-broker-core` first, later client/MCP slices will have to redefine them.
- **Leaving the S01 transition table untouched** — the current table allows closures and shortcuts that do not match legacy lifecycle semantics; parity tests will be ambiguous until this is resolved.
- **Forgetting notification/version bumps on non-claim mutations** — messages, verdicts, counter-patch decisions, and close/requeue paths all need to wake `review-status:${reviewId}` and often `review-queue`, or the wait semantics proven in S01 will silently regress.
- **Hand-porting Python tests 1:1 without adapting to local patterns** — per the loaded `test` skill, keep the project’s existing Vitest harness/style and use the Python tests as behavior references, not as a structural template.

## Open Risks

- The biggest unresolved design choice is whether S02 treats `submitted` as the TS alias for legacy `in_review`, or reintroduces Python’s literal status string. This choice changes every contract and test file touched by the slice.
- Legacy Python activity/status payloads include more metadata (`intent`, `category`, `project`, `current_round`, `verdict_reason`) than the current TS `ReviewSummary`. Planner should decide explicitly which of these are required for S02 parity vs deferred to later slices.
- Requeue semantics after proposer follow-up currently depend in Python on reviewer reservation/future pool behavior. S02 can preserve the review-state part now, but some reservation details may need to stay compatible with S03 reviewer-lifecycle work.
